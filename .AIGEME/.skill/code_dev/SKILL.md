## 产物存放规则

所有产物（`ARCHITECTURE.md`、代码文件、`QA_REPORT.md` 等）**默认存放在角色自己的工作区目录**（`workspace/`），除非用户明确指定其他路径。每个文件路径使用工作区绝对路径。

---

## 阶段零：思维对齐（对齐前不得进入下一阶段）

在开始编码前，你**必须**先向用户输出你的理解，对齐后再进入 Architecture Design。

输出内容：
- 你对需求的理解（功能列表、边界）
- 技术选型理由（语言、框架、库）
- 为什么选这个方案而不是其他方案

在确认无误后，进入 Architecture Design。

---

## 阶段一：Architecture Design

产出 `ARCHITECTURE.md`，包含：

1. **整体架构** — 模块划分和数据流
2. **模块职责** — 每个模块做什么、不做什么
3. **接口设计** — 函数签名、类结构、配置项
4. **关键设计决策** — 为什么这么做、替代方案
5. **数据流** — 核心业务流程的调用链

---

## 阶段二：Coding + 静态检查

**编码原则：**

1. 每次只写 1-2 个文件
2. 写完一个文件就做静态检查（`mypy` / `ruff` / `flake8` 等）
3. 报错立刻修
4. 所有文件写完后，确保代码结构完整、可运行

**编码时的 Attention：**

- 大语言模型倾向于在有坑的地方忽视细节
- 检查以下易错点：
  - 类型注解是否与实际返回类型一致
  - 异步函数是否有 `await`
  - 文件路径拼接是否用 `Path` / `os.path.join`
  - 异常处理是否覆盖了 IO / 网络等易错操作
  - 配置项默认值是否合理
  - 多线程/异步是否有竞态条件

---

## 阶段三：QA 审核

产出 `QA_REPORT.md`：

1. **功能清单对照** — 逐条检查是否实现
2. **代码质量评分**（1-5 分）：可读性、可维护性、健壮性、性能
3. **改进建议** — 按优先级列出
4. **结论** — 通过 / 有条件通过 / 不通过

---

## 各阶段输出

| 阶段 | 产出 |
|------|------|
| 对齐 | 用户确认 |
| Architecture | `ARCHITECTURE.md` |
| Coding | 代码文件 + 静态检查日志 |
| QA | `QA_REPORT.md` |
